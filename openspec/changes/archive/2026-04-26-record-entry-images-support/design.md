## Context

当前前端的图片相关链路分成三段但没有打通：`CreateRecordDialog` 中的 `ImageUploadAnalyzer` 负责 AI 识图辅助填写，`AiAnalysisPreview` 负责多食物识别后批量创建记录，`BarcodeScanner` 负责条码识别并查询营养信息。三条链路都能拿到源图片或商品图片，但在最终调用 `calorieService` 创建记录时，没有把图片 URL 写入后端记录模型的 `images` 字段。

后端已经提供两类可复用能力：一类是 `POST /gateway/ai/image-nutrition`，用于上传图片并返回识图结果；另一类是 `POST /storage/presign-upload` + 对象存储 `PUT` 上传，用于获取可公开访问的图片 URL。后端卡路里记录 DTO 与 schema 也已支持 `images?: string[]`。因此本次改动重点不是新增模型，而是把前端图片预处理、上传、记录保存、列表展示与编辑回显统一起来，并把组件里的上传逻辑与展示 UI 解耦。

## Goals / Non-Goals

**Goals:**
- 为饮食记录建立统一的“本地图片 → 预处理 → 上传到对象存储 → 得到 URL → 写入 `images` → 回显展示”的前端链路。
- 将图片上传的状态管理、错误处理、上传动作从现有 UI 组件中拆分出来，形成可复用的 service / hook / 展示组件组合。
- 让手动饮食记录、AI 识图批量保存、条码确认保存都能在创建记录时附带图片 URL。
- 在记录列表与编辑弹窗中展示已保存图片，保证保存后立即回显、刷新后仍可见。

**Non-Goals:**
- 不改动后端存储服务协议、对象存储目录结构或鉴权机制。
- 不新增前端服务端路由、中转 API 或本地缓存层。
- 不扩展运动记录图片上传能力，本次仅覆盖 `type="intake"` 的饮食记录。
- 不在本次变更中实现多图排序、裁剪编辑、删除远端已上传文件等高级媒体管理能力。

## Decisions

### 1. 使用后端对象存储预签名上传，而不是复用 AI 分析接口返回图片地址

决策：新增独立的前端存储上传 service，先请求 `POST /storage/presign-upload`（`businessType=calorie-image`），再由浏览器直接 `PUT` 图片到对象存储，最后将返回的 `publicUrl` 写入记录 DTO 的 `images` 字段。

原因：
- AI 识图接口当前职责是“上传并分析”，其返回体不包含可复用的图片 URL，不能直接承担记录附件持久化。
- 条码与手动上传场景同样需要图片保存，走统一存储接口可以避免把“是否需要 AI 分析”与“是否需要保存记录图片”耦合到一起。
- 对象存储链路已由后端封装好业务目录与公开 URL 生成逻辑，前端只需遵循契约。

备选方案：
- 复用 `gateway/ai/image-nutrition` 并要求后端返回上传 URL：会把分析与持久化强绑定，条码和手动场景无法复用。
- 直接把 base64 或本地对象 URL 存进记录：不能跨设备、刷新后失效，也不符合后端 DTO 的 URL 约束。

### 2. 抽离“图片附件”领域层：service 负责上传，hook 负责状态，UI 只负责展示

决策：将现有 `ImageUploadAnalyzer` 中混杂的文件选择、预处理、分析与展示逻辑拆开，形成三层：
- `services/storageService.ts`：封装预签名请求与 PUT 上传。
- `hooks` 或组件内复用逻辑：管理本地文件、预览 URL、上传状态、已保存 URL、清理行为。
- 独立展示组件：负责上传卡片、缩略图、操作按钮、错误提示。

原因：
- 当前 `CreateRecordDialog` 和 `AiAnalysisPreview` 都需要访问图片文件与上传状态，如果继续把行为塞在单个组件里，会导致扫码、AI 识图和手动录入之间重复实现。
- UI / 逻辑分离更符合仓库现有约定，后续也便于在头像、健康建议等其他图片场景中复用存储上传能力。

备选方案：
- 仅在 `ImageUploadAnalyzer` 内部追加上传行为：能工作，但扫码和批量 AI 场景无法复用，而且组件职责会进一步膨胀。

### 3. 记录创建时使用“懒上传”策略，只在用户确认保存时真正上传图片

决策：用户选择或识别图片后，前端先只保留预处理后的 `File` 与本地预览，等用户点击“保存记录”或“确认添加”时再执行上传并将 URL 写入 DTO。

原因：
- 避免用户只做试识别或扫码后取消，却产生无引用的远端垃圾文件。
- 保持表单编辑体验，允许用户在保存前反复修改标题、营养值和餐别。
- 与现有 `handleSubmitRecord`、`handleBatchSubmitRecords`、`handleConfirmBarcodePreview` 的确认时机一致，利于最小化改动。

备选方案：
- 选图后立即上传：交互更快，但会额外制造未保存记录的孤儿文件，需要后端清理策略支持。

### 4. AI 批量保存与扫码保存统一写入 `images[0]`

决策：本次所有单张来源图片都写成记录的 `images` 数组首项；AI 批量识图保存多条记录时，同一张源图可以复用到每一条生成记录的 `images[0]`。

原因：
- 后端模型是 `string[]`，但本次场景均为单图来源，使用数组首项即可满足当前协议并保留将来扩展空间。
- AI 批量识图的多条记录来自同一张图片，重复引用同一 URL 能保留识别来源，且不需要额外上传多次。

备选方案：
- 仅给第一条 AI 记录附图：会导致同批次其他识别结果缺失来源图，编辑与回显体验不一致。

### 5. 列表中展示轻量缩略图，编辑弹窗展示完整预览并支持保留已有 URL

决策：在记录列表中仅在存在 `images[0]` 时显示小尺寸缩略图；编辑弹窗打开已有记录时，优先回显远端 URL，并允许用户保留原图或替换为新图。

原因：
- 列表场景以快速浏览为主，缩略图足够提示记录附图存在，不应显著拉高卡片高度。
- 编辑弹窗是最合适的图片替换入口，可以同时处理已有 URL 和新选择本地文件的状态切换。

备选方案：
- 只在弹窗展示，不在列表展示：无法满足“存在图片时支持显示图片”的核心需求。

## Risks / Trade-offs

- [重复上传同一图片到多条 AI 记录] → 通过“同一批量保存仅上传一次，再复用同一 URL”降低冗余。
- [用户选择图片但未保存即关闭弹窗] → 采用懒上传策略，仅生成本地预览，不产生远端垃圾文件。
- [对象存储未配置 `publicUrl`] → 前端在上传 service 中对 `publicUrl` 为空做失败处理，阻止写入无效图片地址。
- [编辑记录替换图片时状态复杂] → 明确区分“已有远端 URL”“待上传本地文件”“已清空图片”三种状态，避免隐式覆盖。
- [新增缩略图影响列表性能] → 仅渲染首图，限制尺寸并使用浏览器原生懒加载/受控样式，不引入重型图片组件。

## Migration Plan

1. 新增前端对象存储上传 service 与图片附件复用状态层。
2. 扩展 `CreateCalorieEntryDto` / 相关构造逻辑，允许创建与编辑时提交 `images`。
3. 改造手动饮食录入、AI 批量识图、条码确认保存三条链路，在保存前执行图片上传并写回 URL。
4. 改造记录列表与编辑回显，验证新增图片记录与历史无图记录都能正常展示。
5. 若上线后发现上传链路异常，可回滚前端变更；后端接口无 schema 迁移需求，旧记录仍兼容。

## Open Questions

- 条码场景若同时存在“用户上传的扫码原图”和商品接口返回的 `imageUrl`，默认应优先保存哪张图片？当前建议优先用户上传原图，商品图仅作预览兜底。
- 编辑弹窗是否需要显式“移除已保存图片”按钮？当前设计建议支持，以便把 `images` 更新为空数组。
