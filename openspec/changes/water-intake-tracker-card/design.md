## Context

首页当前右栏已经承载 `ProfileSummaryCard` 和 `DailyCalorieCalendar` 两个独立卡片，整体结构适合继续追加健康数据型卡片。相邻的 backend 已实现 `water` 模块，接口形式为 `GET /water?startDate=&endDate=` 查询日期范围饮水量，以及 `PUT /water` 覆盖设置 `{ date, amount }`。前端现有模式已经明确：服务层统一走 `lib/api.ts`，全局业务状态优先用 Zustand，页面层只做组装，复杂模块需要拆出逻辑层与展示层。

## Goals / Non-Goals

**Goals:**
- 为首页增加一个独立的今日饮水卡片，作为快速记录入口
- 基于 backend `water` 接口实现今日饮水读取和覆盖更新
- 使用 Zustand 集中管理今日饮水状态、错误、加载与提交流程
- 将饮水模块拆分为逻辑控制层和纯展示视图层
- 采用 `500ml` 一杯的点击计数交互，保证记录成本足够低

**Non-Goals:**
- 不实现饮水历史日历、周/月统计或趋势分析
- 不改造 backend 水记录接口的请求/响应结构
- 不把饮水逻辑混入现有 `useCalorieTracker` 或卡路里 store
- 不引入新的 UI 或状态管理依赖

## Decisions

### 1. 使用独立 `waterService` 对接 `/water` 接口

**选择**: 新增独立饮水服务，而不是复用 `calorieService` 或散落在组件中直接请求。

**理由**:
- backend 已有独立 `water` 领域，前端按业务域拆分更符合现有结构
- `GET /water` 是日期范围查询，首页今日数据只需要用当天日期同时填充 `startDate` 和 `endDate`
- `PUT /water` 为覆盖更新，恰好适配杯子计数器“点到哪杯就设置到哪杯”的交互语义

**备选方案**:
- 复用 `calorieService`：会混淆业务边界，不利于后续扩展饮水历史
- 组件内直接请求：违背当前仓库“组件不直接调用 fetch”的约定

### 2. 使用独立 `waterIntakeStore` 管理今日状态

**选择**: 新增一个只关心“今天”的 Zustand store，包含 `amount`、`loading`、`submitting`、`error` 与 `fetchToday`、`setAmountByCup` 等 action。

**理由**:
- 用户需求聚焦“获取和设置今日饮水”，无需把日期切换能力加入首版
- 与现有 `calorieStore`、`healthAdviceStore` 的集中状态模式一致，页面接入成本低
- 保持饮水逻辑独立，避免污染卡路里主链路和 `useCalorieTracker`

**备选方案**:
- 把饮水状态并入 `calorieStore`：会让 store 职责膨胀，且饮水不属于卡路里条目列表
- 使用局部 `useState`：请求、回滚和跨组件复用会变得分散

### 3. 交互采用“点击某杯即设置总量”的覆盖模型

**选择**: 杯子计数器中的每个杯子代表累计总量，点击第 N 杯即把今日饮水量设置为 `N * 500ml`。

**理由**:
- backend `PUT /water` 是覆盖更新，不需要维护增量语义
- 用户既可快速增加，也可通过点击较前的杯子快速修正为更小的杯数
- 比“每次仅 +500ml”更容易回退，避免额外减号按钮或重置入口

**备选方案**:
- 每次点击只做 `+500ml`：减少实现分支，但用户修正输入不方便
- 使用滑块或数字输入：不符合用户希望的“按杯喝水”心智模型

### 4. 模块拆为容器组件和纯视图组件

**选择**: 将饮水模块拆为一个负责接 store/token 的容器组件，以及一个只接收 props 的杯子视图组件。

**理由**:
- 满足“逻辑封装和 UI 视图拆分”的明确要求
- 便于后续单独复用杯子计数器或替换首页布局，而不改业务逻辑
- 纯视图层更容易编写状态场景与交互测试

**备选方案**:
- 单文件组件：初期更快，但状态、样式和交互会耦合在一起，不利于维护

### 5. 非 500ml 倍数数据保留真实值文案，视觉按整杯展示

**选择**: 当 backend 返回的历史数据不是 `500ml` 的整数倍时，卡片文案展示真实 `amount`，杯子高亮按 `Math.floor(amount / 500)` 计算。

**理由**:
- backend DTO 只要求 `amount >= 0`，前端不能假设历史数据必定是 `500` 的倍数
- 文案保留真实值可避免误导用户，整杯视觉仍保持交互一致性
- 用户下一次点击某杯后，后端数据将被自然归一化到整杯值

**备选方案**:
- 直接四舍五入杯数：会造成杯子数和真实毫升文本不一致
- 增加半杯/部分填充动画：超出首版最小实现范围

## Risks / Trade-offs

- **[后端历史数据非整杯值]** -> 保留真实毫升文本，杯子仅表达完整杯数，避免视觉夸大
- **[首页右栏高度继续增长]** -> 饮水卡片保持信息密度简洁，仅展示核心数字与杯子交互，减少垂直占用
- **[快速连点导致重复提交]** -> store 提供 `submitting` 状态，提交过程中禁用交互并在失败时回滚
- **[今日饮水与首页日期选择器语义不同]** -> 在卡片文案中明确标注“今日饮水”，避免用户误解为随左侧日期切换

## Migration Plan

1. 先落类型与服务层，确保 `/water` 接口封装独立可用。
2. 实现 `waterIntakeStore` 与容器层，完成读取、覆盖更新与失败回滚。
3. 实现独立饮水 Card 和杯子视图，并接入首页右栏。
4. 完成后以首页手动交互和类型检查验证，不涉及数据迁移。
5. 如需回滚，只需移除首页卡片接入并删除饮水前端模块，不影响后端数据结构。

## Open Questions

- 首版杯子计数器默认展示多少个杯位更合适，建议以 8 杯（`4000ml`）作为常规上限，超出时再补充扩展策略。
- 是否需要在卡片中补充推荐饮水目标文案，这不影响当前实现，但会影响最终视觉信息密度。
