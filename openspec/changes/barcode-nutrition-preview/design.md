## Context

当前首页的扫码入口会打开 `BarcodeScanner`，组件完成图片识别后仅通过提示框展示条码内容，没有把条码号继续转成可落库的饮食数据。与此同时，现有首页记录链路已经具备统一的提交入口：`useCalorieTracker` 暴露 `handleSubmitRecord`，底层通过 `calorieStore -> calorieService` 写入记录。

这意味着本次变更的核心不是“再做一个扫码器”，而是在扫码识别成功之后补齐一段业务编排：查询后端条码营养接口，展示商品预览，再由用户确认是否创建一条 `source="barcode"` 的饮食记录。

当前仓库内没有后端条码接口的实现或契约文件，因此前端需要在服务层做一层响应适配，把后端返回的数据收敛为统一的预览模型，避免接口字段变化影响 UI 与创建记录逻辑。

## Goals / Non-Goals

**Goals:**
- 扫码识别成功后，立即基于条码号查询后端商品营养信息
- 在创建记录前展示结构化预览，至少覆盖商品名、条码号、热量和主要营养成分
- 用户确认后直接创建一条条码来源的饮食记录，并刷新首页数据
- 条码无匹配、查询失败、用户取消三类分支都具备明确反馈和返回路径
- 将后端接口契约差异限制在独立 service 和类型映射层中

**Non-Goals:**
- 不修改条码识别算法本身，不新增新的扫码引擎或图像预处理策略
- 不在预览弹层中支持手动编辑营养值
- 不处理条码商品的多规格换算、单位换算或份量换算规则扩展
- 不改造现有 AI 图片分析和手动创建记录流程

## Decisions

### 1. 流程编排放在首页业务状态，而不是塞进 `BarcodeScanner` 内部

**选择**：`BarcodeScanner` 只负责输出识别成功的条码结果；查询营养信息、展示预览、确认创建记录放在 `useCalorieTracker` 与首页组合层管理。

**替代方案**：
- 在 `BarcodeScanner` 内直接请求后端并提交记录
- 在 `BarcodeScanner` 内同时渲染扫描视图和预览视图

**理由**：扫码器是识别组件，不应承载记录创建业务。把查询和确认逻辑放在业务层可以复用认证、刷新列表、错误提示能力，也能避免 `BarcodeScanner.tsx` 继续膨胀。

### 2. 新增独立 `barcodeNutritionService`，屏蔽后端接口路径与字段差异

**选择**：新增条码营养查询 service，内部负责调用后端 API 并将响应映射为前端统一的 `BarcodeNutritionPreview` 模型。

**替代方案**：
- 直接在组件中 `fetch`
- 复用 `calorieService`，把条码查询也塞进同一 service 文件

**理由**：当前仓库约定组件不直接调用 `fetch`，服务按业务域拆分。条码查询与卡路里记录 CRUD 不是同一职责，拆成独立 service 更符合单一职责，也能吸收后端接口未完全确认的风险。

### 3. 预览采用单独确认弹层，确认后直接创建记录

**选择**：在扫码成功并查到商品后，打开条码营养预览弹层，提供“添加到记录”和“取消”两个主要动作；点击确认后直接调用现有记录提交链路创建饮食记录。

**替代方案**：
- 预填到 `CreateRecordDialog` 再让用户二次编辑
- 查询成功后立即自动创建，不做确认

**理由**：用户请求明确要求先看预览再决定是否添加。直接创建会缺少确认，跳入手动表单又会把扫码流程拉长。预览确认弹层刚好覆盖用户决策点，同时复用已有提交接口。

### 4. 创建记录时保留条码来源元数据

**选择**：确认添加时提交 `type="intake"`、`source="barcode"`、`externalId=<barcode>`，并附带后端返回的营养信息。

**替代方案**：
- 只提交标题和热量，不保留条码号
- 将条码号塞进 description 文本

**理由**：现有类型已经支持 `source` 和 `externalId`，直接复用可以为后续编辑、去重或溯源留出空间，不需要额外改表单协议。

### 5. API 契约不确定性通过“最小预览模型”收敛

**选择**：前端内部只依赖一个最小模型：`barcode`、`name`、`brand?`、`calories`、`nutrition?`、`servingText?`。service 中负责把后端原始响应转换为该模型。

**替代方案**：
- 让 UI 直接消费后端响应字段
- 在类型里完整镜像后端所有字段

**理由**：当前无法在本仓库中验证后端返回结构，过早绑定原始字段会把不确定性扩散到多个组件。先定义稳定的前端消费模型，后续只需调整适配函数。

## Risks / Trade-offs

- **[后端接口路径或字段与预期不一致]** -> 将差异限制在独立 service 的适配函数中，实施时先以真实后端契约补齐映射
- **[条码查无商品]** -> 预览不打开，直接给出“未找到营养信息”的错误提示，并允许用户重新扫码或关闭
- **[查询时间较长导致体验割裂]** -> 在扫码结果与预览之间增加 loading 状态，禁用重复触发
- **[条码商品营养信息不完整]** -> 允许仅展示已有字段，创建记录时只提交存在的营养项，避免因为缺少单个字段阻断流程
- **[直接添加不可编辑]** -> 这是当前有意的流程收敛；如后续需要微调，可在下一次迭代增加“预览后继续编辑”分支

## Open Questions

- 后端条码营养查询 API 的确切路径、请求方法和响应字段命名需要在实施前以真实后端项目为准
- 若后端返回按每 100g 或每份计算的营养值，前端是否需要展示单位说明，当前先按后端直接可展示文本处理
