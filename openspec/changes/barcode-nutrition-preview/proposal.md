## Why

当前条码扫描能力只解决了“识别出条码号”，没有继续把条码号映射成可用的食品营养信息。用户在扫码成功后仍需要手动录入标题、热量和营养成分，导致扫码入口的价值没有真正落到“快速记一条饮食记录”上。

## What Changes

- 在条码扫描成功后，调用后端条码营养查询 API，根据条码号获取商品名称、品牌、份量信息、热量和营养成分
- 新增条码营养预览界面，在正式创建记录前展示查询结果，并允许用户确认添加或取消返回
- 仅在用户确认后，才将条码查询结果转换为饮食记录草稿并进入现有记录创建/保存流程
- 为条码营养查询新增独立服务封装与类型定义，统一处理认证、错误提示和空结果状态
- 在现有首页记录创建链路中接入扫码成功后的预览确认流程，保留手动录入和 AI 图片分析现有能力

## Capabilities

### New Capabilities
- `barcode-nutrition-preview`: 条码扫描成功后查询商品营养信息并展示确认预览，覆盖查询、结果展示、确认添加与取消返回流程

### Modified Capabilities
- `calorie-management`: 首页新增记录流程在扫码成功后进入条码营养预览，并在用户确认后创建条码来源的饮食记录

## Impact

- **组件**：`BarcodeScanner.tsx` 将从“只负责识别”扩展为“识别后交接结果”；首页和新增记录相关组件需要承接条码预览确认流程
- **服务层**：新增条码营养查询 service，封装后端 API 调用与响应解析
- **类型**：新增条码商品与营养预览类型，明确预览模型与创建记录 DTO 的映射
- **状态管理**：`useCalorieTracker` 或相邻业务状态需要管理扫码结果、预览弹层开关、确认后提交行为
- **后端依赖**：依赖现有条码营养查询 API 返回结构化商品营养数据；若条码无匹配结果，需要前端提供明确提示与回退路径
